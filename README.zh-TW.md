# Image Collection v2

[English](README.md) | 繁體中文 | [简体中文](README.zh-CN.md)

透過 IMEI 號碼從 NAS 共享資料夾批次收集裝置影像的桌面工具。使用 Electron、React 及 Liquid Glass UI 主題開發。**v1.5.10 — 無日期的 MR 收集。**

操作人員匯入稽核清單 — 建議包含 IMEI、機器及日期欄位以獲得最快結果 — 選擇要搜尋的機器資料夾，並以可設定的整理方式將匹配的影像資料夾匯出至本機目標位置。

---

## 快速開始

1. **安裝** — 執行 `Image Collection v2 Setup X.X.X.exe` 並依照提示操作。
2. **設定來源** — 點擊共享資料夾旁的**瀏覽**，選擇 NAS 根目錄（例如 `Z:\`）。機器資料夾將自動填入。
3. **匯入稽核清單** — 在稽核面板點擊**瀏覽**，或拖放 CSV / XLSX 檔案。**為獲得最佳結果，請包含 IMEI、機器及日期欄位** — Smart Search 會自動偵測這些欄位、自動勾選相關機器資料夾、自動填入日期範圍，並將搜尋時間從數分鐘縮短至數秒。
4. **確認偵測結果** — 檢查稽核面板中的綠色標記，確認機器及日期欄位已被偵測。確認來源面板中僅勾選了相關的機器資料夾。若偵測失敗或資料不準確，請關閉 Smart Search 以回復為僅限 IMEI 模式。
5. **配置設定** — 選擇動作、影像類型、整理模式，並設定目標位置。
6. **搜尋** — 點擊**開始搜尋**。匹配結果將即時串流至結果表格。
7. **匯出** — 檢視結果後，點擊**匯出結果**。進度顯示於狀態列。

## 功能

| 功能 | 說明 |
|---|---|

| **日期範圍篩選** | 將搜尋限制在特定日期區間（YYYYMMDD 資料夾名稱） |
| **掃描索引篩選** | 依掃描索引編號篩選（`_1`、`_2`、`_3` 等） |
| **搜尋與匯出記錄** | 診斷搜尋記錄（路徑決策、各機器計數、備援）＋詳細的逐檔匯出記錄；各保留最近 3 份 |
| **缺少 IMEI 報告** | 檢視並儲存在 NAS 上未找到的稽核 IMEI 清單 |
| **深色 / 淺色主題** | 兩種模式均採用 Liquid Glass 主題；跨工作階段保存 |
| **三語** | 英文、繁體中文及簡體中文 — 透過來源面板的語言按鈕切換 |
| **設定保存** | 所有偏好設定透過 electron-store 儲存，並於下次啟動時還原 |

## 來源配置

### 共享資料夾

輸入或瀏覽至 NAS 掛載點（例如 `Z:\`、`\\NAS_Lonestar\share`）。工具會掃描第一層子目錄並填入資料夾切換格。

> **不限於 NAS。** PC 可存取的任何磁碟機皆可搜尋 — 包括連接至 SMART Grade PC 或 RPC 的本機硬碟 — 只要資料夾結構符合標準的 `機器 → YYYYMMDD → IMEI_Index` 階層即可。

- **機器資料夾**（名稱符合 `M` + 數字，如 `M8`、`M10`）預設為勾選。
- **公用資料夾**（`audits`、`crackimages`、`GRR Images`、`version_control`）預設為不勾選。
- `#recycle` 及 `$RECYCLE.BIN` 始終隱藏。

使用**全選 / 取消全選**進行批次切換。**重新整理**可重新掃描來源以更新新增或移除的子資料夾。

### 多來源管理

將常用的 NAS 路徑儲存為命名來源：

1. 瀏覽至 NAS 根目錄。
2. 點擊 **[+]** 並輸入名稱（例如「NAS Lonestar」）。
3. 按下 Enter 儲存。

透過下拉選單切換來源。每個來源各自記憶其資料夾切換狀態。點擊 **[-]** 可移除來源（因為此操作無法復原，會顯示確認提示）。

首次啟動時，先前儲存的單一路徑設定會自動轉移為命名來源。

### 日期範圍

將搜尋限制在特定日期區間的資料夾：

- **起始 / 結束日期** — 僅掃描 `YYYYMMDD` 名稱落在此範圍內的資料夾。兩者皆留空則搜尋所有日期。
- 若起始日期晚於結束日期，會顯示警告（不會有任何匹配結果）。

## 稽核清單匯入

拖放或瀏覽以匯入稽核檔案。

### 建議方式：Smart Search（IMEI + 機器 + 日期）

收集影像的最快方式是在稽核檔案（CSV 或 Excel）中包含 **IMEI**、**機器**及**日期**欄位。當三個欄位皆存在時，Smart Search 會直接前往 NAS 上每個 IMEI 的確切資料夾 — 將搜尋時間從數分鐘縮短至數秒。

**建議的稽核檔案格式：**

| IMEI | 機器 | 日期 |
|---|---|---|
| 350308317018557 | M14 | 2026-05-14 |
| 350448886316469 | M21 | 2026-05-14 |
| 351101754222413 | M22 | 2026-05-14 |

欄位標題可以是任意文字 — 解析器透過資料模式識別欄位，而非依據標題名稱。欄位順序無關。額外欄位會被忽略。IMEI 欄位始終透過 15 位數字值來識別。

**機器欄位** — 可識別以下名稱格式：

| 檔案中的值 | 標準化為 |
|---|---|
| M8, M10 | M8, M10（完全匹配） |
| M08, M-8, M-08 | M8（移除前導零 / 連字號） |
| SG-M16, LAX-M08 | M16, M8（移除站點前綴） |
| Machine 8 | M8（擷取數字） |
| 08, 8 | M8（純數字） |

**日期欄位** — 可識別以下日期格式：

| 格式 | 範例 |
|---|---|
| YYYYMMDD | 20260515 |
| YYYY-MM-DD | 2026-05-15 |
| YYYY/MM/DD | 2026/05/15 |
| MM/DD/YYYY | 05/15/2026 |
| MM-DD-YYYY | 05-15-2026 |
| M/D/YYYY | 5/15/2026 |
| M/D/YY | 5/15/26 |
| M/D/YY HH:MM | 5/14/26 11:57（自動移除時間部分） |

任何尾隨的時間部分（例如 `11:57`、`11:57:00` 或 ISO `T` 分隔符）在解析前會被移除。對於模糊的日期格式（MM/DD 與 DD/MM），解析器會取樣所有資料列以判斷正確的解讀方式。完全模糊時預設使用 MM/DD（美國格式）。

**匯入後**，稽核面板以色碼標記顯示偵測品質：
- **綠色**：所有資料列解析成功
- **橘色**：80% 以上資料列解析成功（部分值無法識別）
- **紅色**：低於 80% 解析成功（欄位偵測可能不可靠）

**自動填入日期範圍**：當偵測到日期欄位時，來源面板的起始 / 結束日期欄位會自動填入稽核資料中最早及最晚的日期。這可縮小搜尋範圍，並提供日期解析正確的視覺確認。如有需要，可手動調整日期。

**自動選取機器資料夾**：當偵測到機器欄位時，來源面板會自動勾選稽核資料中引用的機器資料夾，並取消勾選其餘資料夾。例如，若稽核檔案僅包含 M14、M21 和 M22 的條目，則僅這三個資料夾會被勾選。這可避免掃描無關的機器，加快廣泛掃描的備援速度。若未偵測到機器欄位，所有機器資料夾將維持先前的切換狀態。

**部分提示**：稽核檔案不需要包含全部三個欄位。Smart Search 會根據可用的欄位進行調整：

| 可用欄位 | 搜尋行為 |
|---|---|
| IMEI + 機器 + 日期 | 直接資料夾查找 — 最快（數秒） |
| 僅 IMEI + 機器 | 廣泛掃描但限縮至每個 IMEI 提示的機器 |
| 僅 IMEI | 跨所有已選機器進行完整廣泛掃描（最慢） |

缺少值或值無法識別的 IMEI 會自動回復至下一層廣泛掃描 — 不會遺漏任何結果，這些 IMEI 只是搜尋較慢。

**MR PASS / FAIL 收集**仰賴機器 + 日期提示：它會開啟每個裝置的確切資料夾 `Machine/{date}/{IMEI}/` 並取出其中的影像 — 不會列舉日期資料夾，因此無論該資料夾多大都能瞬間完成。

若偵測到的欄位不準確，請關閉 **Smart Search** 以回復為僅限 IMEI 模式。

### 替代方案：僅限 IMEI 匯入

若您的稽核資料僅包含 IMEI（無機器或日期欄位），工具會回復為廣泛 NAS 掃描 — 搜尋所有已選機器中的每個日期資料夾。此方式可行但速度明顯較慢。

| 格式 | 行為 |
|---|---|
| **CSV** | 透過取樣前 20 列自動偵測 IMEI 欄位。略過標題列。支援逗號、Tab 及分號分隔符。 |
| **XLSX / XLS** | 讀取第一個工作表。自動偵測 IMEI 欄位。略過標題列。 |
| **TXT** | 每行一個值。每行視為潛在的 IMEI。TXT 檔案無法使用 Smart Search（單一欄位）。 |

**IMEI 驗證**：必須為恰好 15 位數字。無效項目及重複項目會被計數並報告。

> **OneDrive 檔案：** 若您的稽核檔案位於設定為「僅限線上」的 OneDrive 資料夾中，首次讀取可能會在 OneDrive 下載檔案時卡住。本工具現在會在 20 秒後逾時，並顯示清楚的訊息而非無限掛起。若要徹底避免此問題，請在檔案總管中右鍵點擊該檔案 →**一律保留在此裝置上**（等待綠色勾號出現），或將稽核檔案保存在一般的本機資料夾中。

## 設定參考

### 動作

| 值 | 行為 |
|---|---|
| **複製**（預設） | 將匹配的資料夾複製至目標位置。來源資料不受影響。 |
| **移動** | 傳輸資料夾並在複製後**刪除來源**。啟用前會顯示確認對話框。下次啟動時，動作始終重設為複製以確保安全。 |

> 移動模式**不適用於** MR 影像收集 — 只會複製那一張 MR 影像（`.png`）（永不移動），因此每個裝置的資料夾保持完整。

### 影像類型

| 值 | 行為 |
|---|---|
| **全部**（預設） | 匯出匹配資料夾中的所有影像檔案。 |
| **BMP** | 僅匯出 `.bmp` 檔案。 |
| **JPEG** | 僅匯出 `.jpg` / `.jpeg` 檔案。 |

### 整理

控制匯出資料夾在目標位置的排列方式：

| 模式 | 結構 | 範例 |
|---|---|---|
| **平鋪** | 全部放在同一資料夾 | `dest/350002267153742_192/` |
| **依機器** | 依來源機器分組 | `dest/M8/350002267153742_192/` |
| **依日期** | 依掃描日期分組 | `dest/20260515/350002267153742_192/` |
| **機器 > 日期** | 兩層：先機器再日期 | `dest/M8/20260515/350002267153742_192/` |
| **日期 > 機器** | 兩層：先日期再機器 | `dest/20260515/M8/350002267153742_192/` |
| **依 IMEI** | 將同一裝置的所有實例分組 | `dest/350002267153742/M8_20260515_192/` |
| **依型號** | 依裝置型號分組（從 MR 影像解析） | `dest/Apple-iPhone13Pro/350002267153742_192/` |
| **機器 > 型號** | 兩層：先機器再裝置型號 | `dest/M8/Apple-iPhone13Pro/350002267153742_192/` |

**MR 影像匯出**時，每個 IMEI 會有自己的資料夾，內含匹配的 `.png` 檔案：

| 模式 | MR 範例 |
|---|---|
| 平鋪 | `dest/350002267153742/M8_20260515_Samsung-Galaxy_S24.png` |
| 機器 > 日期 | `dest/M8/20260515/350002267153742/Samsung-Galaxy_S24.png` |
| 依 IMEI | `dest/350002267153742/M8_20260515_Samsung-Galaxy_S24.png` |
| 依型號 | `dest/Samsung-Galaxy_S24/350002267153742/M8_20260515_Samsung-Galaxy_S24.png` |
| 機器 > 型號 | `dest/M8/Samsung-Galaxy_S24/350002267153742/20260515_Samsung-Galaxy_S24.png` |

> **始終包含：** 每次匯出都會自動包含每個 IMEI 資料夾中的 `DefectLog.xml` 和 MR 影像（`SG-*.png`），不受影像類型篩選設定影響。這些檔案對稽核和型號識別至關重要。

> **提示：** 整理模式獨立於搜尋 — 您可以在搜尋完成後、匯出前更改模式。匯出將使用您點擊**匯出結果**時所選的模式。

### 重複

| 值 | 行為 |
|---|---|
| **略過**（預設） | 若目標資料夾 / 檔案已存在，不進行變更。 |
| **覆蓋** | 以新的來源資料取代現有的目標資料夾 / 檔案。 |

### 掃描索引

| 值 | 行為 |
|---|---|
| **全部**（預設） | 包含每個掃描索引（`_1`、`_2`、`_3` 等）。 |
| **僅第一個** | 僅 `_1` 項目 — 該裝置在該機器 / 日期的系列掃描中的第一個。 |

### MR PASS / MR FAIL — 模型辨識影像收集

這兩個切換開關會為稽核清單中的每個 IMEI 收集該裝置的**模型辨識（MR）影像** — 當您的稽核檔案是一份由評級系統標記的裝置清單（例如「顏色錯誤」報告），而您只需要每個裝置的 MR 擷取影像時，這是正確的選擇。

**您通常不需要動這些切換開關。** 當稽核含有評級 / 失敗欄位（例如 `Grade-D2C` =「顏色錯誤」）時，工具會偵測到並**自動啟用 MR 收集** — 綠色橫幅會加以確認。（手動開啟任一切換開關的效果相同。）

**運作方式。** 被標記（「顏色錯誤」）的裝置儲存在一個**完全以其 IMEI 命名**的資料夾中 — `Machine/{date}/{IMEI}/`，**沒有 `_index` 後綴** — 內含一張以時間戳記命名的影像 `.png`。工具會**直接開啟該確切路徑**並取出 `.png`：

```
Machine/{date}/{IMEI}/2026-06-04-10-17-03.png
```

依路徑開啟已知資料夾永遠不會列舉龐大的父層日期資料夾（數千個子資料夾 — 過於龐大，無法透過網路列舉），因此**無論資料夾大小都能瞬間收集**。`ModelRecogImages/` 從不被掃描。

- **依型號整理：** 裝置型號來自稽核的 `Model` 欄位（移除顏色，`Apple-iPhone11-Purple` → `Apple-iPhone11`），因此**依型號**及**機器 → 型號**能正確分組這些影像。收集本身從不需要型號欄位 — 缺少它時，只有這兩種模式會回退至 `Unknown` 資料夾；其他所有模式皆不受影響。
- **每個裝置只匯出那一張 `.png`** — MR 收集在啟用期間會取代標準的整資料夾收集。
- 若某裝置的資料夾有找到但其中不含 `.png`，則會回報為缺少（並計入記錄）。
- **萬無一失：** 確切路徑查找會在每次 Smart Search 時執行，因此即使 MR 從未啟用、稽核也沒有評級欄位，這些影像仍會被收集。

> 稽核清單本身就是篩選條件 — 沒有「顏色錯誤」切換開關。載入清單並搜尋；工具會根據稽核的欄位自行配置。

> 完整掃描（一般、非 MR）裝置的儲存方式不同 — 位於 `{IMEI}_{index}` 資料夾中，內含掃描 JPG 以及一張 `SG-*.png` — 並由標準影像收集處理，而非這些切換開關。

### 僅 AI 影像

啟用時，僅從每個匹配的 IMEI 資料夾匯出 `FD/` 子資料夾內容（AI 偵測影像）。資料夾根目錄的標準掃描影像會被排除。

停用時，會匯出完整的 IMEI 資料夾，包含 `FD/` 作為子資料夾。

### 目標位置

匯出影像寫入的本機資料夾。瀏覽或輸入路徑。跨工作階段保存。

## 結果

搜尋完成後，結果面板顯示：

- **摘要列** — 找到的 IMEI 總數 vs. 稽核清單中的總數、經過時間。
- **狀態圓點** — 綠色（完整）、橘色（不完整：檔案數量明顯低於中位數）、紅色（缺少：NAS 上未找到）。
- **可排序表格** — 點擊任何欄位標題（IMEI、機器、日期、索引、檔案）進行升冪 / 降冪排序。
- **檔案明細** — 每個匹配項目顯示總檔案數，並以 `(Xb Yj)` 格式顯示 BMP 及 JPEG 數量。
- **檢視缺少的 IMEI** — 切換表格以顯示未匹配的稽核項目。
- **儲存缺少的 IMEI** — 將缺少清單匯出為 `.txt` 或 `.csv` 檔案。

### 不完整偵測

當匹配項目的檔案數量低於所有匹配項目檔案數量中位數的 50% 時，會被標記為不完整（橘色圓點）。這可以發現存在但影像數量異常少的 IMEI 資料夾。

### 錯誤回饋

如果搜尋或匯出遭遇錯誤（網路中斷、目標位置無法存取、權限問題），會出現錯誤橫幅顯示具體訊息。橫幅在下次操作開始時自動清除，也可手動關閉。

如果搜尋期間有資料夾無法存取，狀態列會顯示計數（例如「3 個存取錯誤」）。這可以區分真正缺少的 IMEI 與因網路或權限問題而無法檢查的 IMEI。

## 版本歷史

### v1.5.10 — 無日期的 MR 收集（2026-06-12）

讓 MR 影像收集即使在稽核清單缺少「日期」欄位（甚至完全沒有提示欄位）時也能運作，因此無論檔案多麼不完整，操作員都能取得影像 — 而且絕不會退回到會在 NAS 上逾時的緩慢資料夾列舉。

- **日期範圍的確切路徑探測。** 當裝置沒有逐一的日期時，MR 收集現在會以確切路徑探測所選日期範圍內的每一天 — 對每一天嘗試 `Machine/{date}/{IMEI}/` — 並在裝置實際受測的那一天取出影像。沒有機器提示的裝置則會在每個所選的機器資料夾中探測。由於每次查找都是直接開啟路徑（從不列出目錄），即使範圍很寬也能保持快速且對 NAS 安全：130 部裝置在 16 天範圍內約為 2,000 次瞬間查找。具有完整「機器 + 日期」提示的裝置仍是單次查找（維持不變 — 含日期的正式清單收集方式與先前完全相同）。
- **可搭配純 IMEI 清單運作。** 只要您選取機器資料夾並設定開始／結束日期範圍，即使稽核僅有 IMEI，MR 收集也能找到裝置。若沒有任何可依據的資訊（無日期範圍且無「日期」欄位），現在會以明確提示告知您需要提供什麼。
- **修正了誤導性的稽核警告。** 試算表中的空白尾端列不再計入提示列總數，因此乾淨的 130 部裝置檔案會顯示「130/130」，而非錯誤地警告數百個「IMEI 的機器值無法辨識」。

### v1.5.9 — 稽核清理與 MR 統一（2026-06-11）

一次唯讀的多代理稽核促成了一輪無用程式碼移除、一項架構簡化，以及防禦性的解析強化。對於目前有效稽核清單的收集方式沒有任何改變 — 正式清單仍以相同方式解析（402 列 → 397 個唯一值、5 個重複），且確切路徑的 MR 收集維持不變。

- **將 MR 收集統一至確切路徑引擎。** 已淘汰被取代的 `SG-*.png` 廣泛掃描 MR 分支 — 那是第二套分歧的實作，只能找到 Type-A 的 `{IMEI}_index` 資料夾，並**遺漏**以裸 `{IMEI}` 命名、含時間戳記 `.png` 的顏色錯誤裝置。MR 現在一律以確切路徑開啟每部裝置的資料夾，而且（新增）只要稽核含有「機器」與「日期」便會執行，**與 Smart Search 開關無關**。兩個欄位皆無的清單現在會顯示明確提示，而非執行找不到 MR 影像的緩慢掃描。
- **移除稽核發現的無用程式碼**：未使用的 `expandDateRange` 輔助函式、未使用的 `ExportLogger` 值別名、從未讀取的 `entryCount` 欄位、無用的 `lastDestination` 設定預設值，以及只寫不讀的每部裝置 `grade` 提示（MR 稽核訊號仍由評級欄位推導，只是不再逐一儲存）。
- **解析資料完整性強化**：Excel 檔案現在會解析含有最多 IMEI 的工作表（而非一律取第一個），因此封面／備註工作表無法遮蔽資料；標題偵測也會辨識欄位關鍵字，使外觀像數字的標題不會被誤判為資料列；而無法判定的 M/D 與 D/M 日期順序會在稽核摘要中標記為「assumed」。
- **穩健性**：「儲存缺少的 IMEI」寫入失敗時現在會顯示錯誤，而非靜默失敗。
- **準確性**：程式碼註解、型別文件，以及 MR PASS / MR FAIL 工具提示皆已修正為描述目前的確切路徑行為（先前有數處仍引用舊的 `SG-*.png` 做法）。

### v1.5.8 — 強化稽核解析（2026-06-11）

針對真實稽核檔案的防禦性解析變更，可**防止 IMEI 被靜默遺失**，且不改變乾淨檔案的讀取方式（正式清單仍以相同方式解析 — 402 列、397 個唯一值、標記 5 個重複）：

- **含分隔符的 IMEI 可被復原** — 像 `35-998765-432109-8` 或 `3539 7910 2606 579` 這類值會在驗證前先正規化（移除空格／破折號），因此格式化的欄位不再丟失每一列。
- **科學記號的 IMEI 可被還原** — 當 Excel 將 15 位數的 IMEI 儲存為數字時，會顯示為 `3.50308E+14`；解析器現在會讀取儲存格的真實數值並寫回完整的 `350308317018557`（精確無誤，因為 15 位數可容於 JS 整數）。
- **可解讀 Excel 序列日期** — 匯出為純數字的日期儲存格（例如 `46180`）會被轉換為真實的日曆日期，並限定範圍於約 2009–2064 年，因此其他欄位的普通整數絕不會被誤判。所有既有的日期格式（`YYYYMMDD`、`YYYY-MM-DD`、`M/D/YY`，含或不含時間）皆維持不變。
- **可解碼 UTF-16 / BOM 文字檔** — 儲存為「Unicode 文字」（UTF-16 LE/BE）或帶有 UTF-8 BOM 的 CSV 或 TXT 現在能被正確讀取，而非讀成無法比對任何項目的亂碼位元組。
- **重複的 IMEI 保留最近一次測試** — 當同一個 IMEI 出現多次（重新測試的裝置）時，其機器／日期／型號提示取自**最新**的日期，同時每個重複項仍如先前一樣在結果中被標記。
- **大型清單（5 萬筆以上）** 以相同的完整性處理 — 絕不以正確性換取速度；大型清單只是耗時較久。

### v1.5.7 — 萬無一失的收集（2026-06-11）

- **即使在「錯誤」的設定下也能運作。** 每次 Smart Search 現在都會先執行即時的**確切路徑探測**：它直接開啟每個裝置的 `Machine\{date}\{IMEI}\` 資料夾並抓取影像。儲存為單純 `{IMEI}` 資料夾的裝置（顏色錯誤 / MR 上傳）**無論是否啟用 MR、也無論稽核是否有評級欄位，都會被收集** — 因此即使操作人員不了解這些切換開關，仍能取得其影像。標準的 `{IMEI}_index` 資料夾僅針對探測未命中的 IMEI，由既有的列舉流程找出，因此這些 IMEI 不會比以往更慢。
- 所有整理模式（平鋪、依機器、依日期、機器→型號、依型號……）皆適用於 MR 匹配 — 不再要求必須使用平鋪。
- 影像收集從不依賴型號欄位；缺少該欄位時，只有依型號 / 機器→型號會回退至 `Unknown` 資料夾（其他所有模式皆不受影響）。

### v1.5.6 — 黏性標題修復（2026-06-11）

- **結果表格標題不再有殘影** — 黏性的欄位標題（IMEI／機器／日期／索引／檔案）原本帶有半透明背景，因此捲動的資料列會透出。現在已改為不透明，資料列得以乾淨地通過其後方。適用於匹配與缺少 IMEI 兩個表格；其他捲動區域（資料夾格、設定下拉選單）經審視後本已正確。

### v1.5.5 — 稽核驅動的 MR 收集（2026-06-11）

讓 MR 收集變得萬無一失，並新增完全由稽核檔案驅動的型號分組：

- **自動偵測 MR 稽核** — 當稽核含有評級 / 失敗欄位（例如 `Grade-D2C` =「顏色錯誤」）時，工具會將其辨識為 MR 收集清單，並**自動啟用 MR 收集**，無論 MR PASS / MR FAIL 切換開關為何。橫幅會加以確認。操作人員不再需要知道該設定哪個切換開關。
- **機器 → 型號（及依型號）現在適用於這類清單** — 由於精簡的 MR 資料夾在檔名中不帶型號，裝置型號改從稽核的 `Model` 欄位解析（移除顏色，例如 `Apple-iPhone11-Purple` → `Apple-iPhone11`）並用於整理。匯出會落在 `dest/M12/Apple-iPhone11/{IMEI}/…`。
- 每個 IMEI 的**評級**也會一併擷取，為日後針對不同失敗類型量身打造奠定基礎。

### v1.5.4 — 確切路徑的 MR 收集（2026-06-11）

一次 NAS 檢查揭露了實際的結構：MR（「顏色錯誤」）裝置擁有一個**完全以 IMEI 命名**的資料夾 — `Machine/{date}/{IMEI}/`，**沒有 `_index` 後綴** — 內含一張**以時間戳記命名的 `.png`**（而非 `SG-*.png`）。先前版本搜尋的是 `{IMEI}_*` + `SG-*.png`，但這兩者對這些裝置都不存在，因此每次查找都回傳「File Not Found」。

- **確切路徑的 MR 收集** — MR 模式現在**直接**開啟 `Machine/{date}/{IMEI}/`（此路徑完全可從稽核的 IMEI + 機器 + 日期得知）並取出其中的 `.png`。沒有萬用字元、沒有 `{IMEI}_*`，而且 — 關鍵在於 — **不會列舉龐大的父層日期資料夾**，因此無論資料夾大小都能瞬間完成，並免於先前版本因並行飽和而逾時的問題。
- 無論 `.png` 名稱為何都會被取用（這些裝置為時間戳記命名，完整掃描資料夾則為 `SG-*`）。

### v1.5.3 — 伺服器端資料夾查找（2026-06-11）

正式環境的 `Machine/{date}/` 資料夾含有極多的 IMEI 子資料夾，以致於即使只是透過 SMB *列舉* 其中一個也會逾時（15 秒）— 這正是搜尋回傳 0 且每個資料夾都報錯的原因。解決之道是根本不去列舉它們：

- **伺服器端 IMEI 查找** — 工具現在針對每個裝置請求 NAS 只回傳匹配的 `{IMEI}_*` 資料夾（`dir /b /a:d {IMEI}_*`）。萬用字元在 NAS 上套用，因此能在**不列舉目錄**的情況下找到資料夾 — 無論其含有多少子資料夾都很快。它在子行程中執行，因此緩慢的 NAS 不再會鎖死 Node 的檔案執行緒池（這也是先前搜尋後 UI 卡頓、包括「檢視記錄」卡頓的原因）。
- **讀取錯誤不再卡住** — 資料夾讀取失敗不再觸發全歷史的廣泛掃描；受影響的 IMEI 會回報為缺少並記錄錯誤。
- **同步搜尋記錄** — 每一行都立即寫入磁碟，因此即使在搜尋途中或取消後也始終可讀（先前緩衝的記錄看起來是空的），並會計時緩慢的查找。

### v1.5.1 — MR 可靠性與診斷（2026-06-10）

源於一次真實的「顏色錯誤」收集，397 張影像中回傳了 0 張。已找出並修復根本原因：

**MR 收集 — 現在快速且可靠：**
- MR 收集（啟用 MR PASS 或 MR FAIL **任一個**）現在使用**快速 IMEI 資料夾查找**，只從每個裝置的資料夾中取出那張 `SG-*.png` MR 影像。MR 影像就位於每個 IMEI 資料夾內，與掃描影像並列，因此收集可在數秒內完成。
- 它不再列舉 `ModelRecogImages/{Brand-Model}/` 資料夾 — 這些資料夾累積了數萬個檔案，透過網路掃描會逾時（這正是先前回傳空白結果與 UI 卡住的根本原因）。
- 結果會依影像的型號名稱標記為 PASS（綠色）/ FAIL（紅色）；只會匯出 `SG-*.png`。
- 「顏色錯誤」只能從稽核清單得知，永遠無法從 NAS 得知 — 因此清單就是篩選條件；沒有「顏色錯誤」切換開關。

**日期處理：**
- 自動填入的**結束日期現在是最後測試日期 + 1 天**，因此在接近午夜時測試的裝置（資料夾換到隔天）不再被遺漏。
- 缺少 / 不存在的資料夾（ENOENT）被視為良性，不再計為掃描錯誤。

**診斷：**
- **搜尋現在會被記錄**（`search-*.log`），包含完整請求、定向 vs 探索的路徑決策、每機器資料夾計數、備援轉換，以及結果摘要，讓短促的搜尋也能一目了然地診斷。**檢視記錄**會開啟記錄資料夾。

**可靠性：**
- **過期狀態修復** — 日期範圍會在搜尋時從已提交狀態讀取，且機器自動選取以內容為鍵。更改設定 / 切換開關後，不再需要重新上傳稽核檔案搜尋才能運作。
- **OneDrive 安全讀取** — 20 秒逾時並顯示清楚的「仍在下載」訊息，而非無限掛起。

**新增：**
- **機器 → 型號**整理模式（`dest/M8/Apple-iPhone13Pro/IMEI_index/`）。MR 匹配現在會帶有解析出的型號名稱。

### v1.4.0 — 強化版本（2026-05-28）

全面可靠性與安全性審計。識別並修復 30 個問題，涵蓋所有嚴重程度等級。

**關鍵修復：**
- **移動模式安全性** — 當影像類型篩選啟用時（會破壞未選取的檔案）、未複製任何檔案時、或複製失敗時，阻止來源刪除
- **原子覆寫** — 覆寫前備份目標資料夾，失敗時自動還原，防止中斷匯出導致資料遺失
- **日誌防當機保護** — 串流錯誤處理器防止未處理的例外導致應用程式當機

## 從原始碼建置

### 前置需求

- [Node.js](https://nodejs.org/) v18 或更新版本
- npm（隨 Node.js 附帶）

### 開發

```bash
git clone <repo-url>
cd Image-Collection-v2
npm install
npm run dev          # 啟動並啟用熱重載
```

### 正式建置

```bash
npm run build        # 編譯 main + preload + renderer
npm run pack         # 建置 + 建立未封裝目錄（快速，用於測試）
npm run dist         # 建置 + 建立 NSIS 安裝程式（.exe）
```

安裝程式輸出至 `release/Image Collection v2 Setup X.X.X.exe`。

### 其他指令

| 指令 | 說明 |
|---|---|
| `npm run typecheck` | 對 main 及 renderer 執行 TypeScript 類型檢查 |
| `npm run lint` | 使用 ESLint 檢查原始碼 |
| `npm run format` | 使用 Prettier 格式化原始碼 |

## 安裝

執行 NSIS 安裝程式。它將：

- 安裝至 `C:\Program Files\Image Collection v2\`
- 建立**桌面捷徑**及**開始功能表捷徑**
- 在**新增或移除程式**中註冊，以便完整解除安裝
- 可選擇在安裝完成後啟動應用程式（完成頁面的核取方塊）

安裝後不需要管理員權限。應用程式將設定及記錄儲存於 `%APPDATA%/Image Collection v2/`。

### 解除安裝

使用**新增或移除程式**（設定 > 應用程式 > 已安裝的應用程式）。解除安裝程式會移除所有檔案、捷徑、登錄機碼及應用程式資料 — 零殘留。

## 系統需求

- **作業系統**：Windows 10 或更新版本（64 位元）
- **NAS 連線**：NAS 共享必須已掛載或可透過 UNC 路徑存取（例如 `\\NAS_Lonestar\share`）
- **磁碟空間**：應用程式約需 200 MB；目標磁碟需要足夠的空間存放匯出影像
- **無執行階段相依性**：Node.js 已與 Electron 應用程式打包在一起
