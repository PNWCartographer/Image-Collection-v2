# Image Collection v2

[English](README.md) | [繁體中文](README.zh-TW.md) | 简体中文

通过 IMEI 号码从 NAS 共享文件夹批量收集设备图像的桌面工具。使用 Electron、React 及 Liquid Glass UI 主题开发。**v1.5.10 — 无日期的 MR 收集。**

操作人员导入审计清单 — 建议包含 IMEI、机器及日期列以获得最快结果 — 选择要搜索的机器文件夹，并以可配置的整理方式将匹配的图像文件夹导出至本地目标位置。

---

## 快速开始

1. **安装** — 运行 `Image Collection v2 Setup X.X.X.exe` 并按照提示操作。
2. **设置来源** — 点击共享文件夹旁的**浏览**，选择 NAS 根目录（例如 `Z:\`）。机器文件夹将自动填充。
3. **导入审计清单** — 在审计面板点击**浏览**，或拖放 CSV / XLSX 文件。**为获得最佳结果，请包含 IMEI、机器及日期列** — Smart Search 会自动检测这些列、自动选中相关机器文件夹、自动填充日期范围，并将搜索时间从数分钟缩短至数秒。
4. **验证检测结果** — 检查审计面板中的绿色标记，确认机器及日期列已被检测。确认来源面板中仅选中了相关的机器文件夹。若检测失败或数据不准确，请关闭 Smart Search 以回退到仅限 IMEI 模式。
5. **配置设置** — 选择动作、图像类型、整理模式，并设置目标位置。
6. **搜索** — 点击**开始搜索**。匹配结果将实时流式显示在结果表格中。
7. **导出** — 查看结果后，点击**导出结果**。进度显示在状态栏中。

|---|---|

| **深色 / 浅色主题** | 两种模式均采用 Liquid Glass 主题；跨会话保存 |
| **三语** | 英文、繁体中文及简体中文 — 通过来源面板的语言按钮切换 |

## 来源配置

### 共享文件夹

输入或浏览至 NAS 挂载点（例如 `Z:\`、`\\NAS_Lonestar\share`）。工具会扫描第一层子目录并填充文件夹切换网格。

> **不限于 NAS。** PC 可访问的任何驱动器均可搜索 — 包括连接至 SMART Grade PC 或 RPC 的本地硬盘 — 只要文件夹结构符合标准的 `机器 → YYYYMMDD → IMEI_Index` 层级即可。

- **机器文件夹**（名称匹配 `M` + 数字，如 `M8`、`M10`）默认为选中。
- **实用文件夹**（`audits`、`crackimages`、`GRR Images`、`version_control`）默认为不选中。
- `#recycle` 及 `$RECYCLE.BIN` 始终隐藏。

使用**全选 / 取消全选**进行批量切换。**刷新**可重新扫描来源以更新新增或移除的子文件夹。

### 多来源管理

将常用的 NAS 路径保存为命名来源：

1. 浏览至 NAS 根目录。
2. 点击 **[+]** 并输入名称（例如"NAS Lonestar"）。
3. 按下 Enter 保存。

通过下拉菜单切换来源。每个来源各自记忆其文件夹切换状态。点击 **[-]** 可移除来源（因为此操作无法撤消，会显示确认提示）。

首次启动时，先前保存的单一路径设置会自动迁移为命名来源。

### 日期范围

将搜索限制在特定日期区间的文件夹：

- **起始 / 结束日期** — 仅扫描 `YYYYMMDD` 名称落在此范围内的文件夹。两者均留空则搜索所有日期。
- 若起始日期晚于结束日期，会显示警告（不会有任何匹配结果）。

## 审计清单导入

拖放或浏览以导入审计文件。

### 推荐方式：Smart Search（IMEI + 机器 + 日期）

收集图像的最快方式是在审计文件（CSV 或 Excel）中包含 **IMEI**、**机器**及**日期**列。当三列均存在时，Smart Search 会直接前往 NAS 上每个 IMEI 的确切文件夹 — 将搜索时间从数分钟缩短至数秒。

**推荐的审计文件格式：**

| IMEI | 机器 | 日期 |
|---|---|---|
| 350308317018557 | M14 | 2026-05-14 |
| 350448886316469 | M21 | 2026-05-14 |
| 351101754222413 | M22 | 2026-05-14 |

列标题可以是任意文字 — 解析器通过数据模式识别列，而非依据标题名称。列顺序无关。额外列会被忽略。IMEI 列始终通过 15 位数字值来识别。

**机器列** — 可识别以下名称格式：

| 文件中的值 | 标准化为 |
|---|---|
| M8, M10 | M8, M10（完全匹配） |
| M08, M-8, M-08 | M8（去除前导零 / 连字符） |
| SG-M16, LAX-M08 | M16, M8（去除站点前缀） |
| Machine 8 | M8（提取数字） |
| 08, 8 | M8（纯数字） |

**日期列** — 可识别以下日期格式：

| 格式 | 示例 |
|---|---|
| YYYYMMDD | 20260515 |
| YYYY-MM-DD | 2026-05-15 |
| YYYY/MM/DD | 2026/05/15 |
| MM/DD/YYYY | 05/15/2026 |
| MM-DD-YYYY | 05-15-2026 |
| M/D/YYYY | 5/15/2026 |
| M/D/YY | 5/15/26 |
| M/D/YY HH:MM | 5/14/26 11:57（自动去除时间部分） |

任何尾随的时间部分（例如 `11:57`、`11:57:00` 或 ISO `T` 分隔符）在解析前会被去除。对于模糊的日期格式（MM/DD 与 DD/MM），解析器会采样所有数据行以判断正确的解读方式。完全模糊时默认使用 MM/DD（美国格式）。

**导入后**，审计面板以颜色标记显示检测质量：
- **绿色**：所有数据行解析成功
- **橙色**：80% 以上数据行解析成功（部分值无法识别）
- **红色**：低于 80% 解析成功（列检测可能不可靠）

**自动填充日期范围**：当检测到日期列时，来源面板的起始 / 结束日期字段会自动填入审计数据中最早及最晚的日期。这可缩小搜索范围，并提供日期解析正确的视觉确认。如有需要，可手动调整日期。

**自动选择机器文件夹**：当检测到机器列时，来源面板会自动选中审计数据中引用的机器文件夹，并取消选中其余文件夹。例如，若审计文件仅包含 M14、M21 和 M22 的条目，则仅这三个文件夹会被选中。这可避免扫描无关的机器，加快广泛扫描的回退速度。若未检测到机器列，所有机器文件夹将维持先前的切换状态。

**部分提示**：审计文件不需要包含全部三列。Smart Search 会根据可用的列进行调整：

| 可用列 | 搜索行为 |
|---|---|
| IMEI + 机器 + 日期 | 直接文件夹查找 — 最快（数秒） |
| 仅 IMEI + 机器 | 广泛扫描但限缩至每个 IMEI 提示的机器 |
| 仅 IMEI | 跨所有已选机器进行完整广泛扫描（最慢） |

缺失值或值无法识别的 IMEI 会自动回退到下一层广泛扫描 — 不会丢失任何结果，这些 IMEI 只是搜索较慢。

**MR PASS / FAIL 收集**依赖机器 + 日期提示：它会打开每个设备的确切文件夹 `Machine/{date}/{IMEI}/` 并取出其中的图像 — 不会枚举日期文件夹，因此无论该文件夹多大都能瞬间完成。

若检测到的列不准确，请关闭 **Smart Search** 以回退到仅限 IMEI 模式。

### 备选方案：仅限 IMEI 导入

若您的审计数据仅包含 IMEI（无机器或日期列），工具会回退为广泛 NAS 扫描 — 搜索所有已选机器中的每个日期文件夹。此方式可行但速度明显较慢。

| 格式 | 行为 |
|---|---|
| **CSV** | 通过采样前 20 行自动检测 IMEI 列。跳过标题行。支持逗号、Tab 及分号分隔符。 |
| **XLSX / XLS** | 读取第一个工作表。自动检测 IMEI 列。跳过标题行。 |
| **TXT** | 每行一个值。每行视为潜在的 IMEI。TXT 文件无法使用 Smart Search（单列）。 |

**IMEI 验证**：必须为恰好 15 位数字。无效条目及重复条目会被计数并报告。

> **OneDrive 文件：** 若您的审计文件位于设置为"仅在线"的 OneDrive 文件夹中，首次读取可能会在 OneDrive 下载文件时卡住。本工具现在会在 20 秒后超时，并显示清晰的消息而非无限挂起。若要彻底避免此问题，请在文件资源管理器中右键点击该文件 →**始终保留在此设备上**（等待绿色勾号出现），或将审计文件保存在普通的本地文件夹中。

## 设置参考

### 动作

| 值 | 行为 |
|---|---|
| **复制**（默认） | 将匹配的文件夹复制到目标位置。来源数据不受影响。 |
| **移动** | 传输文件夹并在复制后**删除来源**。启用前会显示确认对话框。下次启动时，动作始终重置为复制以确保安全。 |

> 移动模式**不适用于** MR 图像收集 — 只会复制那一张 MR 图像（`.png`）（永不移动），因此每个设备的文件夹保持完整。

### 图像类型

| 值 | 行为 |
|---|---|
| **全部**（默认） | 导出匹配文件夹中的所有图像文件。 |
| **BMP** | 仅导出 `.bmp` 文件。 |
| **JPEG** | 仅导出 `.jpg` / `.jpeg` 文件。 |

### 整理

控制导出文件夹在目标位置的排列方式：

| 模式 | 结构 | 示例 |
|---|---|---|
| **平铺** | 全部放在同一文件夹 | `dest/350002267153742_192/` |
| **按机器** | 按来源机器分组 | `dest/M8/350002267153742_192/` |
| **按日期** | 按扫描日期分组 | `dest/20260515/350002267153742_192/` |
| **机器 > 日期** | 两层：先机器后日期 | `dest/M8/20260515/350002267153742_192/` |
| **日期 > 机器** | 两层：先日期后机器 | `dest/20260515/M8/350002267153742_192/` |
| **按 IMEI** | 将同一设备的所有实例分组 | `dest/350002267153742/M8_20260515_192/` |
| **按型号** | 按设备型号分组（从 MR 图像解析） | `dest/Apple-iPhone13Pro/350002267153742_192/` |
| **机器 > 型号** | 两层：先机器后设备型号 | `dest/M8/Apple-iPhone13Pro/350002267153742_192/` |

**MR 图像导出**时，每个 IMEI 会有自己的文件夹，包含匹配的 `.png` 文件：

| 模式 | MR 示例 |
|---|---|
| 平铺 | `dest/350002267153742/M8_20260515_Samsung-Galaxy_S24.png` |
| 机器 > 日期 | `dest/M8/20260515/350002267153742/Samsung-Galaxy_S24.png` |
| 按 IMEI | `dest/350002267153742/M8_20260515_Samsung-Galaxy_S24.png` |
| 按型号 | `dest/Samsung-Galaxy_S24/350002267153742/M8_20260515_Samsung-Galaxy_S24.png` |
| 机器 > 型号 | `dest/M8/Samsung-Galaxy_S24/350002267153742/20260515_Samsung-Galaxy_S24.png` |

> **始终包含：** 每次导出都会自动包含每个 IMEI 文件夹中的 `DefectLog.xml` 和 MR 图像（`SG-*.png`），不受图像类型筛选设置影响。这些文件对审计和型号识别至关重要。

> **提示：** 整理模式独立于搜索 — 您可以在搜索完成后、导出前更改模式。导出将使用您点击**导出结果**时所选的模式。

### 重复

| 值 | 行为 |
|---|---|
| **跳过**（默认） | 若目标文件夹 / 文件已存在，不进行更改。 |
| **覆盖** | 以新的来源数据替换现有的目标文件夹 / 文件。 |

### 扫描索引

| 值 | 行为 |
|---|---|
| **全部**（默认） | 包含每个扫描索引（`_1`、`_2`、`_3` 等）。 |
| **仅第一个** | 仅 `_1` 条目 — 该设备在该机器 / 日期的系列扫描中的第一个。 |

### MR PASS / MR FAIL — 模型识别图像收集

这两个切换开关会为审计清单中的每个 IMEI 收集该设备的**模型识别（MR）图像** — 当您的审计文件是一份由评级系统标记的设备清单（例如"颜色错误"报告），而您只需要每个设备的 MR 捕获图像时，这是正确的选择。

**您通常不需要动这些切换开关。** 当审计含有评级 / 失败列（例如 `Grade-D2C` ="颜色错误"）时，工具会检测到并**自动启用 MR 收集** — 绿色横幅会加以确认。（手动开启任一切换开关的效果相同。）

**运作方式。** 被标记（"颜色错误"）的设备存储在一个**完全以其 IMEI 命名**的文件夹中 — `Machine/{date}/{IMEI}/`，**没有 `_index` 后缀** — 内含一张以时间戳命名的图像 `.png`。工具会**直接打开该确切路径**并取出 `.png`：

```
Machine/{date}/{IMEI}/2026-06-04-10-17-03.png
```

按路径打开已知文件夹永远不会枚举庞大的父层日期文件夹（数千个子文件夹 — 过于庞大，无法通过网络枚举），因此**无论文件夹大小都能瞬间收集**。`ModelRecogImages/` 从不被扫描。

- **按型号整理：** 设备型号来自审计的 `Model` 列（去除颜色，`Apple-iPhone11-Purple` → `Apple-iPhone11`），因此**按型号**及**机器 → 型号**能正确分组这些图像。收集本身从不需要型号列 — 缺少它时，只有这两种模式会回退到 `Unknown` 文件夹；其他所有模式均不受影响。
- **每个设备只导出那一张 `.png`** — MR 收集在启用期间会取代标准的整文件夹收集。
- 若某设备的文件夹有找到但其中不含 `.png`，则会报告为缺失（并计入日志）。
- **万无一失：** 确切路径查找会在每次 Smart Search 时执行，因此即使 MR 从未启用、审计也没有评级列，这些图像仍会被收集。

> 审计清单本身就是筛选条件 — 没有"颜色错误"切换开关。加载清单并搜索；工具会根据审计的列自行配置。

> 完整扫描（普通、非 MR）设备的存储方式不同 — 位于 `{IMEI}_{index}` 文件夹中，内含扫描 JPG 以及一张 `SG-*.png` — 并由标准图像收集处理，而非这些切换开关。

### 仅 AI 图像

启用时，仅从每个匹配的 IMEI 文件夹导出 `FD/` 子文件夹内容（AI 检测图像）。文件夹根目录的标准扫描图像会被排除。

禁用时，会导出完整的 IMEI 文件夹，包含 `FD/` 作为子文件夹。

### 目标位置

导出图像写入的本地文件夹。浏览或输入路径。跨会话保存。

## 结果

搜索完成后，结果面板显示：

- **摘要栏** — 找到的 IMEI 总数 vs. 审计清单中的总数、耗时。
- **状态圆点** — 绿色（完成）、橙色（不完整：文件数量明显低于中位数）、红色（缺失：NAS 上未找到）。
- **可排序表格** — 点击任意列标题（IMEI、机器、日期、索引、文件）进行升序 / 降序排序。
- **文件明细** — 每个匹配项显示总文件数，并以 `(Xb Yj)` 格式显示 BMP 及 JPEG 数量。
- **查看缺失的 IMEI** — 切换表格以显示未匹配的审计条目。
- **保存缺失的 IMEI** — 将缺失清单导出为 `.txt` 或 `.csv` 文件。

### 不完整检测

当匹配项的文件数量低于所有匹配项文件数量中位数的 50% 时，会被标记为不完整（橙色圆点）。这可以发现存在但图像数量异常少的 IMEI 文件夹。

### 错误反馈

如果搜索或导出遇到错误（网络中断、目标位置无法访问、权限问题），会出现错误横幅显示具体信息。横幅在下次操作开始时自动清除，也可手动关闭。

如果搜索期间有文件夹无法访问，状态栏会显示计数（例如「3 个访问错误」）。这可以区分真正缺失的 IMEI 与因网络或权限问题而无法检查的 IMEI。

## 日志

每次搜索及导出都会在应用程序的用户数据目录中生成详细的诊断日志文件：

```
%APPDATA%/Image Collection v2/logs/search-YYYYMMDD-HHmmss-mmm.log
%APPDATA%/Image Collection v2/logs/export-YYYYMMDD-HHmmss-mmm.log
```

**搜索日志**准确记录搜索的执行内容 — 当搜索返回的结果比预期少时极为实用：
- **标题** — 模式（标准 vs MR）、IMEI 数量、提示数量、Smart Search 开 / 关、MR PASS/FAIL、扫描索引筛选、日期范围、已选文件夹、NAS 根目录。
- **路径决策** — 搜索是以定向方式（直接文件夹查找）或完整发现执行、构建的日期文件夹数量、每机器计数，以及有多少 IMEI 落入各层回退（及其原因）。
- **摘要** — 找到的匹配数、缺失数、已扫描文件夹数、扫描错误数、耗时。

## 版本历史

### v1.5.10 — 无日期的 MR 收集（2026-06-12）

让 MR 图像收集即使在审计列表缺少"日期"列（甚至完全没有提示列）时也能运作，因此无论文件多么不完整，操作员都能获取图像 — 而且绝不会退回到会在 NAS 上超时的缓慢文件夹枚举。

- **日期范围的确切路径探测。** 当设备没有逐一的日期时，MR 收集现在会以确切路径探测所选日期范围内的每一天 — 对每一天尝试 `Machine/{date}/{IMEI}/` — 并在设备实际受测的那一天取出图像。没有机器提示的设备则会在每个所选的机器文件夹中探测。由于每次查找都是直接打开路径（从不列出目录），即使范围很宽也能保持快速且对 NAS 安全：130 台设备在 16 天范围内约为 2,000 次瞬间查找。具有完整"机器 + 日期"提示的设备仍是单次查找（维持不变 — 含日期的正式清单收集方式与先前完全相同）。
- **可搭配纯 IMEI 列表运作。** 只要您选择机器文件夹并设置开始／结束日期范围，即使审计仅有 IMEI，MR 收集也能找到设备。若没有任何可依据的信息（无日期范围且无"日期"列），现在会以明确提示告知您需要提供什么。
- **修正了误导性的审计警告。** 电子表格中的空白尾部行不再计入提示行总数，因此干净的 130 台设备文件会显示"130/130"，而非错误地警告数百个"IMEI 的机器值无法识别"。

### v1.5.9 — 审计清理与 MR 统一（2026-06-11）

一次只读的多代理审计促成了一轮无用代码移除、一项架构简化，以及防御性的解析强化。对于当前有效审计列表的收集方式没有任何改变 — 正式清单仍以相同方式解析（402 行 → 397 个唯一值、5 个重复），且确切路径的 MR 收集维持不变。

- **将 MR 收集统一至确切路径引擎。** 已淘汰被取代的 `SG-*.png` 广泛扫描 MR 分支 — 那是第二套分歧的实现，只能找到 Type-A 的 `{IMEI}_index` 文件夹，并**遗漏**以裸 `{IMEI}` 命名、含时间戳 `.png` 的颜色错误设备。MR 现在一律以确切路径打开每台设备的文件夹，而且（新增）只要审计含有"机器"与"日期"便会执行，**与 Smart Search 开关无关**。两个列皆无的清单现在会显示明确提示，而非执行找不到 MR 图像的缓慢扫描。
- **移除审计发现的无用代码**：未使用的 `expandDateRange` 辅助函数、未使用的 `ExportLogger` 值别名、从未读取的 `entryCount` 字段、无用的 `lastDestination` 设置默认值，以及只写不读的每台设备 `grade` 提示（MR 审计信号仍由评级列推导，只是不再逐一存储）。
- **解析数据完整性强化**：Excel 文件现在会解析含有最多 IMEI 的工作表（而非一律取第一个），因此封面／备注工作表无法遮蔽数据；标题检测也会识别列关键字，使外观像数字的标题不会被误判为数据行；而无法判定的 M/D 与 D/M 日期顺序会在审计摘要中标记为"assumed"。
- **稳健性**："保存缺失的 IMEI"写入失败时现在会显示错误，而非静默失败。
- **准确性**：代码注释、类型文档，以及 MR PASS / MR FAIL 工具提示皆已修正为描述当前的确切路径行为（先前有数处仍引用旧的 `SG-*.png` 做法）。

### v1.5.8 — 强化审计解析（2026-06-11）

针对真实审计文件的防御性解析变更，可**防止 IMEI 被静默丢失**，且不改变干净文件的读取方式（正式清单仍以相同方式解析 — 402 行、397 个唯一值、标记 5 个重复）：

- **含分隔符的 IMEI 可被恢复** — 像 `35-998765-432109-8` 或 `3539 7910 2606 579` 这类值会在验证前先规范化（移除空格／破折号），因此格式化的列不再丢失每一行。
- **科学计数法的 IMEI 可被还原** — 当 Excel 将 15 位数的 IMEI 存储为数字时，会显示为 `3.50308E+14`；解析器现在会读取单元格的真实数值并写回完整的 `350308317018557`（精确无误，因为 15 位数可容于 JS 整数）。
- **可解读 Excel 序列日期** — 导出为纯数字的日期单元格（例如 `46180`）会被转换为真实的日历日期，并限定范围于约 2009–2064 年，因此其他列的普通整数绝不会被误判。所有既有的日期格式（`YYYYMMDD`、`YYYY-MM-DD`、`M/D/YY`，含或不含时间）皆维持不变。
- **可解码 UTF-16 / BOM 文本文件** — 存储为"Unicode 文本"（UTF-16 LE/BE）或带有 UTF-8 BOM 的 CSV 或 TXT 现在能被正确读取，而非读成无法匹配任何项目的乱码字节。
- **重复的 IMEI 保留最近一次测试** — 当同一个 IMEI 出现多次（重新测试的设备）时，其机器／日期／型号提示取自**最新**的日期，同时每个重复项仍如先前一样在结果中被标记。
- **大型清单（5 万条以上）** 以相同的完整性处理 — 绝不以正确性换取速度；大型清单只是耗时较久。

### v1.5.7 — 万无一失的收集（2026-06-11）

- **即使在"错误"的设置下也能运作。** 每次 Smart Search 现在都会先执行即时的**确切路径探测**：它直接打开每个设备的 `Machine\{date}\{IMEI}\` 文件夹并抓取图像。存储为单纯 `{IMEI}` 文件夹的设备（颜色错误 / MR 上传）**无论是否启用 MR、也无论审计是否有评级列，都会被收集** — 因此即使操作人员不了解这些切换开关，仍能获取其图像。标准的 `{IMEI}_index` 文件夹仅针对探测未命中的 IMEI，由既有的枚举流程找出，因此这些 IMEI 不会比以往更慢。
- 所有整理模式（平铺、按机器、按日期、机器→型号、按型号……）均适用于 MR 匹配 — 不再要求必须使用平铺。

### v1.5.6 — 粘性标题修复（2026-06-11）

- **结果表格标题不再有残影** — 粘性的列标题（IMEI／机器／日期／索引／文件）原本带有半透明背景，因此滚动的数据行会透出。现在已改为不透明，数据行得以干净地通过其后方。适用于匹配与缺失 IMEI 两个表格；其他滚动区域（文件夹网格、设置下拉菜单）经审视后本已正确。

### v1.5.5 — 审计驱动的 MR 收集（2026-06-11）

让 MR 收集变得万无一失，并新增完全由审计文件驱动的型号分组：

- **自动检测 MR 审计** — 当审计含有评级 / 失败列（例如 `Grade-D2C` ="颜色错误"）时，工具会将其识别为 MR 收集清单，并**自动启用 MR 收集**，无论 MR PASS / MR FAIL 切换开关为何。横幅会加以确认。操作人员不再需要知道该设置哪个切换开关。
- **机器 → 型号（及按型号）现在适用于这类清单** — 由于精简的 MR 文件夹在文件名中不带型号，设备型号改从审计的 `Model` 列解析（去除颜色，例如 `Apple-iPhone11-Purple` → `Apple-iPhone11`）并用于整理。导出会落在 `dest/M12/Apple-iPhone11/{IMEI}/…`。

### v1.5.4 — 确切路径的 MR 收集（2026-06-11）

一次 NAS 检查揭露了实际的结构：MR（"颜色错误"）设备拥有一个**完全以 IMEI 命名**的文件夹 — `Machine/{date}/{IMEI}/`，**没有 `_index` 后缀** — 内含一张**以时间戳命名的 `.png`**（而非 `SG-*.png`）。先前版本搜索的是 `{IMEI}_*` + `SG-*.png`，但这两者对这些设备都不存在，因此每次查找都返回"File Not Found"。

- **确切路径的 MR 收集** — MR 模式现在**直接**打开 `Machine/{date}/{IMEI}/`（此路径完全可从审计的 IMEI + 机器 + 日期得知）并取出其中的 `.png`。没有通配符、没有 `{IMEI}_*`，而且 — 关键在于 — **不会枚举庞大的父层日期文件夹**，因此无论文件夹大小都能瞬间完成，并免于先前版本因并发饱和而超时的问题。
- 无论 `.png` 名称为何都会被取用（这些设备为时间戳命名，完整扫描文件夹则为 `SG-*`）。

### v1.5.3 — 服务器端文件夹查找（2026-06-11）

生产环境的 `Machine/{date}/` 文件夹含有极多的 IMEI 子文件夹，以致于即使只是通过 SMB *枚举* 其中一个也会超时（15 秒）— 这正是搜索返回 0 且每个文件夹都报错的原因。解决之道是根本不去枚举它们：

- **服务器端 IMEI 查找** — 工具现在针对每个设备请求 NAS 只返回匹配的 `{IMEI}_*` 文件夹（`dir /b /a:d {IMEI}_*`）。通配符在 NAS 上应用，因此能在**不枚举目录**的情况下找到文件夹 — 无论其含有多少子文件夹都很快。它在子进程中执行，因此缓慢的 NAS 不再会锁死 Node 的文件线程池（这也是先前搜索后 UI 卡顿、包括"查看日志"卡顿的原因）。
- **读取错误不再卡住** — 文件夹读取失败不再触发全历史的广泛扫描；受影响的 IMEI 会报告为缺失并记录错误。
- **同步搜索日志** — 每一行都立即写入磁盘，因此即使在搜索途中或取消后也始终可读（先前缓冲的日志看起来是空的），并会计时缓慢的查找。

源于一次真实的"颜色错误"收集，397 张图像中返回了 0 张。已找出并修复根本原因：

**MR 收集 — 现在快速且可靠：**

## 从源代码构建

### 前置要求

- [Node.js](https://nodejs.org/) v18 或更高版本
- npm（随 Node.js 附带）

### 开发

```bash
git clone <repo-url>
cd Image-Collection-v2
npm install
npm run dev          # 启动并启用热重载
```

### 生产构建

```bash
npm run build        # 编译 main + preload + renderer
npm run pack         # 构建 + 创建未打包目录（快速，用于测试）
npm run dist         # 构建 + 创建 NSIS 安装程序（.exe）
```

安装程序输出至 `release/Image Collection v2 Setup X.X.X.exe`。

### 其他脚本

| 脚本 | 说明 |
|---|---|
| `npm run typecheck` | 对 main 及 renderer 运行 TypeScript 类型检查 |
| `npm run lint` | 使用 ESLint 检查源代码 |
| `npm run format` | 使用 Prettier 格式化源代码 |

## 安装

运行 NSIS 安装程序。它将：

- 安装至 `C:\Program Files\Image Collection v2\`
- 创建**桌面快捷方式**及**开始菜单快捷方式**
- 在**添加或删除程序**中注册，以便完整卸载
- 可选择在安装完成后启动应用程序（完成页面的复选框）

安装后不需要管理员权限。应用程序将设置及日志存储在 `%APPDATA%/Image Collection v2/`。

### 卸载

使用**添加或删除程序**（设置 > 应用 > 已安装的应用）。卸载程序会移除所有文件、快捷方式、注册表项及应用程序数据 — 零残留。

---

## 系统要求

- **操作系统**：Windows 10 或更高版本（64 位）
- **NAS 连接**：NAS 共享必须已挂载或可通过 UNC 路径访问（例如 `\\NAS_Lonestar\share`）
- **磁盘空间**：应用程序约需 200 MB；目标驱动器需要足够的空间存放导出图像
- **无运行时依赖**：Node.js 已与 Electron 应用程序打包在一起

## 已知限制

- 日期范围筛选依据文件夹日期名称（`YYYYMMDD`）运作。不支持日内时间筛选 — 文件夹结构不包含时间编码。
